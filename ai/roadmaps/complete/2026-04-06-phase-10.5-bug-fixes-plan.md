## Part 1: Scheduling Engine Fixes

The solver and conflict checker have fundamental issues that cause incorrect schedules.

### 1.1 Solver schedules tasks in the past

**Problem:** `getWorkingHoursBounds()` in `solver-utils.ts` returns 8 AM as the day start regardless of current time. At 2 PM, the solver places tasks at 8 AM.

**Fix:** After computing working hours bounds, clamp `start` to `max(workingHoursStart, now)`. The solver's `computeFreeSlots()` will then only consider future time slots.

**Files:** `app/lib/agent/solver-utils.ts`, `app/lib/agent/solver.ts`

### 1.2 Conflict checker uses incomplete query filter

**Problem:** `checkForConflicts` in `check-conflicts.ts` only filters blocks where `start_time` falls within today. Blocks that start before today but end during today are invisible.

**Fix:** Change the query to use overlapping range logic: `start_time < endOfDay AND end_time > startOfDay`. Apply the same fix to the schedule API route (`app/api/schedule/route.ts`).

**Files:** `app/lib/agent/tools/check-conflicts.ts`, `app/app/api/schedule/route.ts`

## Part 2: Chat Agent Fixes

The agent lacks visibility into existing schedules and has streaming issues.

### 2.1 Add getScheduledBlocks tool

**Problem:** The agent has no tool to read existing scheduled blocks. When asked "how's my day", it hallucates times instead of reading the actual schedule. This causes the widget/chat to show different times than the web app.

**Fix:** Create a new `get-scheduled-blocks.ts` tool that queries `scheduled_blocks` for today, formats times in the user's timezone using `formatTimeInZone()`, and returns them to the agent. Register it in `agent.ts`. Update the system prompt to instruct the agent to call this tool when asked about existing schedules.

**Files:** `app/lib/agent/tools/get-scheduled-blocks.ts` (new), `app/lib/agent/agent.ts`, `app/lib/agent/system-prompt.ts`

### 2.2 Fix SSE stream dropping last chunk

**Problem:** In `ChatPanel.tsx`, the SSE reader loop exits on `done` without processing the remaining buffer. The last event can be silently lost.

**Fix:** After the while loop, process any remaining content in the buffer before finalizing the message.

**Files:** `app/components/features/ChatPanel.tsx`

## Part 3: Data Integrity Fixes

### 3.1 Task deletion doesn't cascade to scheduled blocks

**Problem:** Deleting a task via the API doesn't remove its scheduled blocks, leaving orphaned entries in the schedule.

**Fix:** In the DELETE handler for `/api/tasks/[id]`, delete matching `scheduled_blocks` rows before deleting the task.

**Files:** `app/app/api/tasks/[id]/route.ts`

### 3.2 Schedule move endpoint has no validation

**Problem:** The PATCH endpoint for moving a block accepts any times without checking: start < end, no conflicts with other blocks, block belongs to user.

**Fix:** Add validation: verify `start_time < end_time`, check for overlapping blocks, and confirm ownership.

**Files:** `app/app/api/schedule/[id]/route.ts`

---

## Part 4: Timezone & Display Fixes

### 4.1 DailyTimeline uses browser timezone instead of user timezone

**Problem:** `DailyTimeline.tsx` uses `date.getHours()` which returns browser-local time, not the user's configured timezone. If they differ, blocks render at wrong positions.

**Fix:** Pass the user's timezone to DailyTimeline and use `Intl.DateTimeFormat` or the existing `formatTimeInZone()` utility for all time calculations.

**Files:** `app/components/features/DailyTimeline.tsx`, `app/app/app/page.tsx`

---

## Part 5: Resilience Fixes

### 5.1 Silent calendar fetch failures

**Problem:** If `fetchAllBusyWindows()` fails (expired token, network error), it returns an empty array silently. The solver then schedules over actual busy times.

**Fix:** Add error handling around the calendar fetch in `create-scheduled-blocks.ts`. If it fails, surface a warning to the user via the agent response rather than proceeding with an empty calendar.

**Files:** `app/lib/agent/tools/create-scheduled-blocks.ts`, `app/lib/calendar/fetchAllBusyWindows.ts`

### 5.2 Widget API base URL hardcoded to localhost

**Problem:** The widget hardcodes `http://localhost:3000` as the API base URL. In production, it can't reach the backend.

**Fix:** Use `process.env.STRIDE_API_URL` with a fallback to the production Vercel URL. Update CSP in `index.html` to allow both localhost and production.

**Files:** `widget/src/main.ts`, `widget/src/preload.ts`, `widget/src/api.ts`, `widget/src/index.html`
